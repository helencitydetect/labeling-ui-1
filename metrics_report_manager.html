proxyS2-4 · Median & P90 handling time per image
Typical time to process one image (P50) and the slow-tail cutoff (P90). The main speed baseline — should decrease as the team calibrates.
Proxy only: Real task duration requires annotation_started_at and annotation_completed_at per label set. These fields are not in the current data. A proxy via queue assigned_at is available but overestimates duration by including idle time.
FormulaP50 / P90 (annotation_completed_at − annotation_started_at)
DB fields neededannotation_started_at and annotation_completed_at on every label_set row
What it will showThe team's real working pace and the slow tail. P90 should be tracked separately to catch the edge cases that block throughput improvements.
RecommendationAdd these timestamps as a priority. Without them, no time-based speed metric can be computed accurately, and throughput targets (e.g. 1 200 images/day) cannot be validated against actual working pace.
not availableS2-5 · Images per hour
Number of completed label sets per active working hour. The main capacity metric — shows whether the daily target is realistic and enables fair comparison across annotators with different schedules.
FormulaCOUNT(completed_images) / SUM(active_hours)
DB fields neededactive_hours per annotator per day, or a work_session table with session_start / session_end
What it will showWhether the 1 200 images/day target is achievable. Without active hours, dividing by calendar time penalises annotators who worked shorter shifts and rewards those who were simply logged in longer.
RecommendationImplement session logging. Even a simple session start/end table would unlock this metric and all per-hour comparisons.
not availableS2-6 · Handling time by label count bucket (0 / 1 / 2–3 / 4–5 / 6+)
Average handling time grouped by image complexity (number of labels). Shows whether slowdowns are caused by complex multi-label images rather than process or training issues.
FormulaP50(duration) GROUP BY label_count_bucket
DB fields neededannotation_started_at, annotation_completed_at
What it will showComplexity jump points — e.g. "images with 4+ labels take 40% longer". Enables separate throughput targets for simple vs complex image types.
RecommendationPrioritise with S2-4. Both require the same two timestamp fields — adding them resolves three speed metrics at once.
not availableS2-7 · Handling time by attribute presence

DB fields neededannotation_started_at, annotation_completed_at, optionally task_complexity_score
What it will showWhich attributes drive the most annotation time. Feeds into guideline prioritisation and tooling decisions.
not availableS2-8 · Incremental time cost by attribute
Extra time added by each attribute compared to images without it. The most actionable speed metric — directly identifies where simplifying a guideline or adding visual examples would save the most time.
FormulaAVG(time | attr present) − AVG(time | attr absent)
DB fields neededannotation_started_at, annotation_completed_at, task_complexity_score
RecommendationHigh ROI metric. Once timestamps are added, focus analysis on the top 5–10 attributes by incremental cost to identify the highest-leverage guideline improvements.
proxyS2-9 · Time by annotator vs team median
Difference between each annotator's typical handling time and the team median. Quickly shows who works faster or slower than the norm, and where to look for training or process gaps.
Proxy: Volume count (label sets per period) is shown above as an indirect proxy. It does not capture speed — an annotator with fewer label sets may have worked fewer hours, not worked more slowly. Real comparison requires per-image duration.
FormulaP50(annotator_duration) − P50(team_duration)
DB fields neededannotation_started_at, annotation_completed_at, active_hours
TargetNear team norm. Deviations >40% from median should be investigated — may indicate a process issue, workload imbalance, or training need.

not availableS3-6 · Attribute precision — TP / (TP + FP)
For reviewed images: how often a positive label by the annotator was confirmed by the admin. High precision means the annotator is not over-calling an attribute. Especially important for high-cost false positives on structural damage attributes.
FormulaTP / (TP + FP) per attribute per annotator
DB fields neededA comparison table storing TP, FP, TN, FN per reviewed image × attribute. Currently these values can be partially reconstructed from raw label diffs but the logic is fragile and must be rebuilt each time.
What it will showWhich attributes are being over-called most often, ranked by false positive rate. Especially important for attributes like roof-structural_extreme where a false positive has real downstream cost.
RecommendationAdd the TP/FP/TN/FN comparison table. This single table unlocks Precision, Recall, and F1 simultaneously — three metrics from one schema addition.
not availableS3-7 · Attribute recall — TP / (TP + FN)
For reviewed images: how often an attribute present in the admin's output was already found by the annotator. Low recall means the annotator is systematically missing a real condition. Critical for high-priority structural attributes where missed labels mean real conditions go undetected.
FormulaTP / (TP + FN) per attribute per annotator
DB fields neededSame TP/FP/TN/FN table as Precision. No additional fields required.
What it will showSystematic miss patterns per attribute — which conditions are most frequently overlooked, and by whom. Current agreement rate approximates this but conflates missed and false positive cases, making it impossible to distinguish the two without the comparison table.
not availableS3-8 · Attribute F1 score — 2·P·R / (P + R)
Balanced summary of precision and recall per attribute. Most useful for cross-attribute comparison and for prioritising which attributes to address in guideline updates — a low-F1 attribute with high review volume represents the highest ROI for calibration effort.
Formula2 × Precision × Recall / (Precision + Recall)
DB fields neededDerived from Precision + Recall — same TP/FP/TN/FN table. No additional fields.
RecommendationZero marginal cost once Precision and Recall are available. Add to the standard reporting dashboard as the single quality summary score per attribute.

not availableS4-3 · Rework rate
Share of images that required additional correction after original annotation — images with more than one label_set version in the revision chain. Shows how much throughput is consumed by rework rather than net-new annotation.
FormulaCOUNT(images with revision chain > 1) / COUNT(DISTINCT image_id)
DB fields neededimage_id consistently populated in all label_set rows (currently absent from the labeler CSV). Also: rework_reason, rework_stage, final_status to distinguish QA correction from other re-processing reasons.
What it will showTrue net productivity. Even a 5% rework rate means 1 in 20 images requires extra time from both the annotator and the reviewer — a significant hidden cost at the current production volume.
RecommendationAdd image_id to all label_set rows. This is the highest-priority missing field — it unlocks rework tracking, revision chains, and avg revisions per item simultaneously.

Section 5 · Annotator Consistency Metrics

Stability and comparability of labeling behaviour

Consistency metrics detect whether annotators interpret and apply labels in a comparable, stable way — not just who is right or wrong but whether the team is calibrated to the same standard. 3 of 5 metrics collected. Day-to-day variance metrics require per-image timestamps.
collectedS5-1 · Avg labels per image by annotator vs team baseline
Detects systematic over- or under-labeling tendencies. Large deviation from the team baseline (1.28) may signal calibration need rather than a one-off error — it means the annotator is consistently applying a different threshold than the rest of the team.
Avg labels per image — baseline 1.28 (same data as S2-3, consistency lens)
Above baseline (≥1.1) Near baseline (0.8–1.1) Below baseline (<0.8) — potential under-labeling
collectedS5-2 · Positive rate by attribute and annotator
How often (% of images) each annotator marks a given attribute as present. Similar values across the team indicate consistent interpretation. Large deviations signal calibration divergence — the annotator is applying a systematically different threshold, not just making random errors.
Positive rate — top 8 attributes · cells highlighted when deviation ≥40% from team avg
Significantly above team avg (>+40%) Significantly below team avg (<−40%) No highlight = within normal range
collectedS5-3 · Deviation from team mean — exact match rate
Each annotator's EMR vs the team average (46%). Positive = above norm; negative = below. Large deviations warrant investigation into root cause before attributing to skill alone — batch composition, working hours, and image type mix all affect the score.

not availableS5-4 · Day-to-day variance in label behaviour — STDDEV
Stability of label density and attribute usage rates over time per annotator. High variance may indicate fatigue, changing interpretation, or inconsistent conditions — rather than consistent performance in either direction.
FormulaSTDDEV(labels_per_image) per annotator per day
DB fields neededDaily per-annotator per-attribute breakdowns with session/workload context to separate genuine inconsistency from normal variation caused by changing input data composition (different image batches on different days).
What it will showWhether an annotator's behaviour is stable over time or drifts. Accumulate 4+ weeks of data before setting a threshold for "acceptable variance".
RecommendationLow implementation cost — requires only daily task-mix snapshots and the session context already needed for speed metrics. Adds longitudinal quality visibility with minimal additional data.
not availableS5-5 · Day-to-day variance in speed — STDDEV
Stability of handling time or throughput over time per annotator. Flags fatigue, unstable pacing, or inconsistent process conditions that would otherwise be hidden behind average figures.
FormulaSTDDEV(images_per_hour) per annotator per day
DB fields neededannotation_started_at, annotation_completed_at for per-image duration; active_hours or session table for daily images-per-hour (not images-per-calendar-day, which conflates working time with schedule differences)
What it will showFlag annotators whose speed fluctuates more than 30–40% day-over-day as candidates for a check-in. Distinguish real speed instability from workload variation caused by different image batches.
RecommendationUnlocked by the same session and timestamp fields required for speed metrics S2-4 through S2-6. No additional DB work needed beyond what is already recommended above.
